**Группа:** 313

## Цель работы
Развить ранее созданный синтаксический анализатор (парсер) до семантического: построить абстрактное синтаксическое дерево (AST) и реализовать проверку контекстно-зависимых условий в соответствии с индивидуальным вариантом курсовой работы.

## Вариант задания
**Тема:** Объявление комплексного числа с инициализацией.

**Пример верной строки:**
```rust
let complex_num2 = num::complex::Complex::new(3.1, -4.2);
```

**Дополнительные примеры корректных строк:**
- `let x = num::complex::Complex::new(1, 2);`
- `let y = num::complex::Complex::new(-5.0, 0.0);`

## Контекстно-зависимые условия (семантические правила)

### 1. Уникальность идентификаторов в области видимости
- В одной области видимости нельзя объявлять два идентификатора с одинаковым именем.
- **Пример ошибки:**
  ```rust
  let x = num::complex::Complex::new(1.0, 2.0);
  let x = num::complex::Complex::new(3.0, 4.0);  // Ошибка: повторное объявление
  ```
- **Ожидаемое сообщение:** `Идентификатор 'x' уже объявлен ранее`

### 2. Совместимость типов аргументов
- Аргументы конструктора `Complex::new` должны быть числовыми литералами (целыми или вещественными).
- **Пример ошибки:**
  ```rust
  let z = num::complex::Complex::new("text", 5.0);  // Ошибка: нечисловой аргумент
  ```
- **Ожидаемое сообщение:** `Аргумент должен быть числовым литералом`

### 3. Допустимые значения числовых литералов
- Целочисленные литералы должны находиться в диапазоне `i32` (от -2 147 483 648 до 2 147 483 647).
- Вещественные литералы должны находиться в диапазоне `f64` (примерно ±1.8e308).
- **Пример ошибки:**
  ```rust
  let big = num::complex::Complex::new(9999999999, 0.0);  // Ошибка: превышение i32
  ```
- **Ожидаемое сообщение:** `Целочисленный литерал 9999999999 выходит за пределы i32`

### 4. Использование объявленных идентификаторов
- Путь `num::complex::Complex::new` считается предопределённым; другие идентификаторы в пути должны быть проверены на существование.
- **Пример ошибки:**
  ```rust
  let w = wrong::path::new(1.0, 2.0);  // Ошибка: неизвестный модуль
  ```
- **Ожидаемое сообщение:** `Идентификатор 'wrong' не объявлен и не является предопределённым`

## Структура AST

### Типы узлов (перечисление `NodeType`)
- `LET_DECL` – объявление переменной (`let`).
- `CALL_EXPR` – вызов функции (конструктора).
- `PATH` – путь из идентификаторов (`num::complex::Complex::new`).
- `INT_LITERAL` – целочисленный литерал.
- `FLOAT_LITERAL` – вещественный литерал.
- `UNARY_EXPR` – унарное выражение (например, `-4.2`).

### Классы узлов (наследуются от `ASTNode`)
- `LetDeclNode` – содержит поля `name` (имя переменной) и `value` (выражение инициализации).
- `CallExprNode` – содержит поля `callee` (путь) и `arguments` (список аргументов).
- `PathNode` – содержит поле `segments` (список строк идентификаторов).
- `IntLiteralNode` – содержит поле `value` (целое число).
- `FloatLiteralNode` – содержит поле `value` (вещественное число).
- `UnaryExprNode` – содержит поля `operator` (строка `"-"`) и `operand` (узел литерала).

# Лабораторная работа 6: Создание внутренней формы представления программы

**Автор:** Kokunov Andrey  
**Группа:** 313  
**Вариант:** 5 (арифметические выражения)

## Цель работы
Изучить методы построения внутреннего представления программы (ВПП) на основе контекстно-свободной грамматики, реализовать синтаксический анализатор методом рекурсивного спуска и преобразовать арифметические выражения в тетрады и ПОЛИЗ.

## Вариант задания
**Язык программирования:** Упрощённый язык арифметических выражений.

**Полное определение КС-грамматики:**
```
E → TA
A → ε | + TA | - TA
T → FB
B → ε | * FB | / FB | % FB
F → num | id | (E)
id → letter {letter | digit | _}
num → digit {digit}
```

**Примеры верных строк:**
- `2 + 3 * 4`
- `(2 + 3) * 4`
- `a + b`
- `10 % 3 + 5`
- `x * (y - 1)`

## Лексические и синтаксические ошибки

### Диаграмма лексера
Лексический анализатор выделяет следующие классы токенов:
- **INTEGER** – последовательность цифр
- **IDENTIFIER** – буква, за которой следуют буквы, цифры или подчёркивания
- **OPERATOR_PLUS** – `+`
- **OPERATOR_MINUS** – `-`
- **OPERATOR_MUL** – `*`
- **OPERATOR_DIV** – `/`
- **OPERATOR_MOD** – `%`
- **LPAREN** – `(`
- **RPAREN** – `)`
- **WHITESPACE** – пробелы, табы, переводы строк
- **ERROR** – недопустимый символ

Схема работы лексера:
1. Пропуск пробельных символов.
2. Распознавание чисел (целых без знака).
3. Распознавание идентификаторов.
4. Распознавание операторов и скобок.
5. При обнаружении недопустимого символа – генерация ошибки.

### Схема рекурсивного спуска для парсера
Каждому нетерминалу грамматики соответствует метод класса `ArithmeticParser`:
- `E()` → вызывает `T()`, затем `A()`
- `A()` → если текущий токен `+` или `-`, потребляет его, вызывает `T()`, генерирует тетраду/ПОЛИЗ, рекурсивно вызывает `A()`
- `T()` → вызывает `F()`, затем `B()`
- `B()` → если текущий токен `*`, `/` или `%`, потребляет его, вызывает `F()`, генерирует тетраду/ПОЛИЗ, рекурсивно вызывает `B()`
- `F()` → если токен `INTEGER` или `IDENTIFIER`, потребляет его, кладёт значение в стек; если токен `(`, потребляет `(`, вызывает `E()`, потребляет `)`

## Внутренняя форма представления программы (тетрады)
Тетрада – четвёрка `(op, arg1, arg2, result)`, где `op` – операция, `arg1`, `arg2` – операнды, `result` – временная переменная (t1, t2, …). Генерация тетрад происходит во время синтаксического анализа с учётом приоритета операций.

### Скриншот с таблицей тетрад для выражения `(2 + 3) * 4`
![Таблица тетрад](assets/screenshots/quadruples.png)

**Полученные тетрады:**
1. `(+, 2, 3, t1)`
2. `(*, t1, 4, t2)`

## ПОЛИЗ (польская инверсная запись)
ПОЛИЗ формируется с использованием стека операций (алгоритм Дейкстры). Для выражений, состоящих исключительно из целых чисел, также вычисляется значение.

### Скриншот с формированием ПОЛИЗ и вычислением арифметического выражения
![ПОЛИЗ и вычисление](assets/screenshots/poliz.png)

**Пример для `2 + 3 * 4`:**
- ПОЛИЗ: `2 3 4 * +`
- Вычисленное значение: `14`

## Реализованные возможности
1. Лексический анализ арифметических выражений с обнаружением недопустимых символов.
2. Синтаксический анализ методом рекурсивного спуска с обработкой ошибок (пропущенный операнд, лишние скобки, неверный символ).
3. Генерация тетрад (внутреннее представление) для корректных выражений.
4. Построение ПОЛИЗ и вычисление значения для выражений из целых чисел.
5. Интеграция в графический интерфейс: новые вкладки, кнопка «Анализ арифметики», отображение результатов.
6. Поддержка операций `+`, `-`, `*`, `/`, `%` с правильным приоритетом.
