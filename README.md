## Цель работы

## Вариант задания

**Пример верной строки:**

```rust
let complex_num2 = num::complex::Complex::new(3.1, -4.2);
```

**Дополнительные примеры корректных строк:**
- `let x = num::complex::Complex::new(1, 2);`
- `let y = num::complex::Complex::new(-5.0, 0.0);`

## Контекстно-зависимые условия (семантические правила)

### 1. Уникальность идентификаторов в области видимости
- В одной области видимости нельзя объявлять два идентификатора с одинаковым именем.
- **Пример ошибки:**
  ```rust
  let x = num::complex::Complex::new(1.0, 2.0);
  let x = num::complex::Complex::new(3.0, 4.0);  // Ошибка: повторное объявление
  ```
- **Ожидаемое сообщение:** `Идентификатор 'x' уже объявлен ранее`

### 2. Совместимость типов аргументов
- Аргументы конструктора `Complex::new` должны быть числовыми литералами (целыми или вещественными).
- **Пример ошибки:**
  ```rust
  let z = num::complex::Complex::new("text", 5.0);  // Ошибка: нечисловой аргумент
  ```
- **Ожидаемое сообщение:** `Аргумент должен быть числовым литералом`

### 3. Допустимые значения числовых литералов
- Целочисленные литералы должны находиться в диапазоне `i32` (от -2 147 483 648 до 2 147 483 647).
- Вещественные литералы должны находиться в диапазоне `f64` (примерно ±1.8e308).
- **Пример ошибки:**
  ```rust
  let big = num::complex::Complex::new(9999999999, 0.0);  // Ошибка: превышение i32
  ```
- **Ожидаемое сообщение:** `Целочисленный литерал 9999999999 выходит за пределы i32`

### 4. Использование объявленных идентификаторов
- Путь `num::complex::Complex::new` считается предопределённым; другие идентификаторы в пути должны быть проверены на существование.
- **Пример ошибки:**
  ```rust
  let w = wrong::path::new(1.0, 2.0);  // Ошибка: неизвестный модуль
  ```
- **Ожидаемое сообщение:** `Идентификатор 'wrong' не объявлен и не является предопределённым`

## Структура AST

### Типы узлов (перечисление `NodeType`)
- `LET_DECL` – объявление переменной (`let`).
- `CALL_EXPR` – вызов функции (конструктора).
- `PATH` – путь из идентификаторов (`num::complex::Complex::new`).
- `INT_LITERAL` – целочисленный литерал.
- `FLOAT_LITERAL` – вещественный литерал.
- `UNARY_EXPR` – унарное выражение (например, `-4.2`).

### Классы узлов (наследуются от `ASTNode`)
- `LetDeclNode` – содержит поля `name` (имя переменной) и `value` (выражение инициализации).
- `CallExprNode` – содержит поля `callee` (путь) и `arguments` (список аргументов).
- `PathNode` – содержит поле `segments` (список строк идентификаторов).
- `IntLiteralNode` – содержит поле `value` (целое число).
- `FloatLiteralNode` – содержит поле `value` (вещественное число).
- `UnaryExprNode` – содержит поля `operator` (строка `"-"`) и `operand` (узел литерала).

# Лабораторная работа 6: Создание внутренней формы представления программы


**Вариант:** 5 (арифметические выражения)

## Цель работы
Изучить методы построения внутреннего представления программы (ВПП) на основе контекстно-свободной грамматики, реализовать синтаксический анализатор методом рекурсивного спуска и преобразовать арифметические выражения в тетрады и ПОЛИЗ.

## Вариант задания
**Язык программирования:** Упрощённый язык арифметических выражений.

**Полное определение КС-грамматики:**
```
E → TA
A → ε | + TA | - TA
T → FB
B → ε | * FB | / FB | % FB
F → num | id | (E)
id → letter {letter | digit | _}
num → digit {digit}
```

**Примеры верных строк:**
- `2 + 3 * 4`
- `(2 + 3) * 4`
- `a + b`
- `10 % 3 + 5`
- `x * (y - 1)`

## Лексические и синтаксические ошибки

### Диаграмма лексера
Лексический анализатор выделяет следующие классы токенов:
- **INTEGER** – последовательность цифр
- **IDENTIFIER** – буква, за которой следуют буквы, цифры или подчёркивания
- **OPERATOR_PLUS** – `+`
- **OPERATOR_MINUS** – `-`
- **OPERATOR_MUL** – `*`
- **OPERATOR_DIV** – `/`
- **OPERATOR_MOD** – `%`
- **LPAREN** – `(`
- **RPAREN** – `)`
- **WHITESPACE** – пробелы, табы, переводы строк
- **ERROR** – недопустимый символ

Схема работы лексера:
1. Пропуск пробельных символов.
2. Распознавание чисел (целых без знака).
3. Распознавание идентификаторов.
4. Распознавание операторов и скобок.
5. При обнаружении недопустимого символа – генерация ошибки.

### Схема рекурсивного спуска для парсера
Каждому нетерминалу грамматики соответствует метод класса `ArithmeticParser`:
- `E()` → вызывает `T()`, затем `A()`
- `A()` → если текущий токен `+` или `-`, потребляет его, вызывает `T()`, генерирует тетраду/ПОЛИЗ, рекурсивно вызывает `A()`
- `T()` → вызывает `F()`, затем `B()`
- `B()` → если текущий токен `*`, `/` или `%`, потребляет его, вызывает `F()`, генерирует тетраду/ПОЛИЗ, рекурсивно вызывает `B()`
- `F()` → если токен `INTEGER` или `IDENTIFIER`, потребляет его, кладёт значение в стек; если токен `(`, потребляет `(`, вызывает `E()`, потребляет `)`

## Внутренняя форма представления программы (тетрады)
Тетрада – четвёрка `(op, arg1, arg2, result)`, где `op` – операция, `arg1`, `arg2` – операнды, `result` – временная переменная (t1, t2, …). Генерация тетрад происходит во время синтаксического анализа с учётом приоритета операций.

## ПОЛИЗ (польская инверсная запись)
ПОЛИЗ формируется с использованием стека операций (алгоритм Дейкстры). Для выражений, состоящих исключительно из целых чисел, также вычисляется значение.

### 2. Работа с AST
Создан тестовый файл `main.c`:
```c
int square(int x) {
    return x * x;
}

int main() {
    int a = 5;
    int b = square(a);
    return b;
}
```


Сгенерировано AST командой:
```
clang -Xclang -ast-dump -fsyntax-only main.c
```

Полученное дерево показывает структуру функций, параметров, операторов и выражений. Например, для функции `square` видно бинарный оператор умножения с двумя операндами-ссылками на параметр `x`.

### 3. Генерация LLVM IR
Сгенерированы два варианта IR:

- **Без оптимизаций (-O0):** `clang -S -emit-llvm -O0 main.c -o main_O0.ll`
- **С оптимизациями (-O2):** `clang -S -emit-llvm -O2 main.c -o main_O2.ll`

#### Сравнение IR

**main_O0.ll** содержит:
- Инструкции `alloca`, `store`, `load` для каждой локальной переменной.
- Явные вызовы функции `square` с передачей аргумента через память.
- Возврат значения через загрузку из памяти.

**main_O2.ll** содержит:
- Функция `square` упрощена до `mul nsw i32 %0, %0` (исчезли `alloca`/`load`/`store`).
- Функция `main` свернута в `ret i32 25` (константное вычисление `square(5)`).
- Удалены все временные переменные и обращения к памяти.

### 4. Оптимизация IR
Применены оптимизации -O1, -O2, -O3. Наблюдаемые изменения:
- **Constant propagation** – значения, известные на этапе компиляции, подставляются непосредственно.
- **Dead code elimination** – удалены неиспользуемые переменные и инструкции.
- **Function inlining** – короткие функции могут быть встроены в вызывающий код.
- **Mem2reg** – продвижение значений из памяти в регистры (SSA-форма).
- **Vectorization** – в случае операций над массивами или структурами используются векторные инструкции.

### 5. Построение CFG
Из-за отсутствия утилиты `opt` в установленной версии LLVM для Windows построение CFG с помощью `-dot-cfg` не удалось выполнить. Однако можно описать структуру CFG на основе IR:

- **Функция square (-O0):** один базовый блок с последовательностью `alloca`, `store`, `load`, `mul`, `ret`.
- **Функция square (-O2):** один базовый блок с `mul` и `ret`.
- **Функция main (-O0):** несколько блоков (инициализация, вызов, возврат).
- **Функция main (-O2):** один блок `ret i32 25`.

Графы потока управления становятся проще после оптимизаций, количество базовых блоков сокращается.

## Индивидуальное задание (вариант)

**Вариант:** Анализ пользовательских операторов для комплексных чисел.

### Исходный код
Создан файл `mycomplex.c` с реализацией структуры `Complex` и операций умножения и сложения:
```c
typedef struct {
    double re;
    double im;
} Complex;

Complex mul(Complex a, Complex b) { ... }
Complex add(Complex a, Complex b) { ... }

int main() {
    Complex z1 = {3.0, 4.0};
    Complex z2 = {1.0, 2.0};
    Complex z3 = add(mul(z1, z2), z1);
    return 0;
}
```

### Анализ AST
AST показывает вложенные вызовы функций, доступ к полям структуры, арифметические операции. Дерево отражает семантику операций умножения и сложения комплексных чисел.

### Генерация и сравнение IR
Сгенерированы IR для уровней оптимизации -O0, -O1, -O2, -O3.

**Наблюдения:**
- **-O0:** Функции `mul` и `add` содержат множество `alloca`, `load`, `store`, отдельные операции над каждым полем.
- **-O3:** Появилась векторизация – операции над полями `re` и `im` объединены в векторные инструкции `<2 x double>`. Функция `main` полностью удалена, так как вычисления не используются (dead code elimination).
- **Инлайн операторов:** Операторы `mul` и `add` не были встроены в `main`, потому что их вызовы находятся внутри выражения, которое в итоге удалено. Если бы результат использовался, компилятор мог бы применить inlining.

### Выводы по индивидуальному заданию
1. LLVM успешно оптимизирует пользовательские операторы, применяя векторизацию, когда это возможно (операции над независимыми полями структуры).
2. Оптимизация dead code elimination удаляет неиспользуемые вычисления, даже если они содержат вызовы пользовательских функций.
3. Встраивание (inlining) пользовательских функций происходит при наличии нескольких вызовов и небольшом размере функции, но в данном примере не произошло из-за удаления всего выражения.
4. Представление структур в IR осуществляется как агрегаты; доступ к полям – через `getelementptr`. При оптимизациях агрегаты могут быть разложены на отдельные скаляры или векторы.

Ответы на 12 контрольных вопросов приведены в отдельном файле [answers.md](answers.md). Кратко:

1. Clang – фронтенд для C/C++, преобразует исходный код в AST и LLVM IR.
2. LLVM – инфраструктура для оптимизации и генерации машинного кода.
